# MYO bridge architecture

This document describes the bridge used to integrate the MYO Armband acquisition process with the main multimodal EEG-sEMG-IMU acquisition environment.

## Purpose

The MYO Armband acquisition process was separated from the main acquisition process because of device and software compatibility requirements.

The original system used two Python environments:

~~~text
Python 3.9  -> MYO sender process
Python 3.10 -> Main acquisition process and MYO receiver
~~~

The MYO sender collected data from the MYO Armband and transmitted the data to the main acquisition process through a TCP JSON-lines socket.

## General architecture

~~~text
MYO Armband
    |
    v
MYO sender process
Python 3.9
    |
    | TCP JSON-lines
    v
MYO receiver process
Python 3.10
    |
    v
Main multimodal acquisition
EEG + sEMG + MYO + events
~~~

## MYO sender

The MYO sender is responsible for:

- Connecting to the MYO Armband.
- Acquiring 8-channel MYO EMG.
- Acquiring orientation data.
- Acquiring accelerometer and gyroscope data.
- Acquiring pose and RSSI information.
- Sending JSON messages to the receiver.
- Receiving START and STOP commands from the receiver.

The sender should remain separate from the main Python environment when MYO SDK compatibility requires it.

## MYO receiver

The MYO receiver is responsible for:

- Opening a TCP server.
- Waiting for the MYO sender connection.
- Sending a START command to the sender.
- Receiving JSON-lines messages.
- Parsing MYO messages.
- Storing received data in a thread-safe buffer.
- Sending a STOP command when the acquisition finishes.

The cleaned public implementation is located in:

~~~text
src/motor_intention/communication/tcp_myo.py
~~~

## Message format

Each MYO message is sent as one JSON object followed by a newline character.

Example message:

~~~json
{
  "timestamp_origin": 0.125,
  "emg": [12, -4, 8, 10, -2, 3, 5, 1],
  "orientation": [1.0, 0.0, 0.0, 0.0],
  "imu": {
    "accelerometer": [0.01, 0.02, 0.98],
    "gyroscope": [0.1, 0.0, -0.1]
  },
  "pose": "rest",
  "rssi": -55
}
~~~

The receiver may add additional timing fields such as:

~~~text
timestamp_received_s
timestamp_rel
~~~

## Commands

The receiver can send commands to the sender using JSON messages.

START command:

~~~json
{"cmd": "START"}
~~~

STOP command:

~~~json
{"cmd": "STOP"}
~~~

## Timestamp notes

Timestamp definitions should be handled carefully.

Recommended fields:

~~~text
timestamp_origin
timestamp_received_s
timestamp_rel
~~~

Suggested interpretation:

- `timestamp_origin`: timestamp generated by the sender.
- `timestamp_received_s`: absolute time when the receiver received the message.
- `timestamp_rel`: receiver-relative time from acquisition start.

For latency estimation, sender and receiver clocks must be defined consistently. If `timestamp_origin` is relative to the sender acquisition start, it should not be directly subtracted from an absolute receiver timestamp.

## Public repository policy

The public repository should not include:

- Hardcoded MYO SDK paths.
- Local user paths.
- Private hardware configuration.
- Raw MYO recordings from participants.
- Participant-identifiable metadata.
- Full private experimental sessions.

## Future migration plan

The next cleaned MYO-related components should be:

~~~text
scripts/acquisition/run_myo_sender.py
configs/acquisition/myo_sender.example.yaml
docs/myo_bridge.md
~~~

The MYO sender script should read hardware-specific settings from configuration files or environment variables instead of hardcoded local paths.

## Public synthetic sender

A public synthetic MYO sender entry point is available at:

~~~text
scripts/acquisition/run_myo_sender.py
~~~

This script currently supports synthetic mode only. It can be used to test the TCP bridge without the MYO Armband or MYO SDK.

The example configuration file is:

~~~text
configs/acquisition/myo_sender.example.yaml
~~~

To show the command-line options, run:

~~~bash
python scripts/acquisition/run_myo_sender.py --help
~~~

Real MYO SDK support will be added later after cleaning the legacy sender code.
